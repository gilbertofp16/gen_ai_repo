# Seven Rules of Power Notes

Source basis: summarized principles from Jeffrey Pfeffer's 7 Rules of Power, complemented by Sameer Bajaj's public notes and the user's personal power notes.

Use these notes as a practical decision framework for reviewing and improving user text.

The goal is not manipulation, arrogance, or fake dominance. The goal is to help the user communicate with more confidence, strategic clarity, visibility, and professional authority.

## Purpose of these notes

These notes are for a skill that reviews pasted text and improves it through a power lens.

The skill should help the user:

- stop making themselves smaller
- remove unnecessary apologies
- communicate achievements clearly
- sound like a peer, not a subordinate
- build a stronger professional brand
- use direct and outcome-focused language
- ask for things without sounding needy
- create visibility
- position themselves around strategic value
- become more respected and easier to act on

## Core philosophy

Power is the ability to get things done.

Power is not only title or formal authority. It also comes from:

- expertise
- visibility
- reputation
- relationships
- timing
- useful information
- confidence
- access to decision-makers
- ability to solve important problems
- ability to make risk visible
- ability to create business value

A message can either increase or reduce power.

The skill should identify power leaks and rewrite the text so the user sounds more credible, decisive, and strategic.

## The seven rules

1. Get out of your own way
2. Break the rules strategically
3. Appear powerful
4. Build a powerful brand
5. Network relentlessly
6. Use your power
7. Success excuses almost everything

## Main power question

Do not ask only:

"Am I being authentic?"

Ask:

"Am I expressing my thinking in a way that others can respect and act on?"

Authenticity without control can leak insecurity.

Confidence without genuineness can feel fake.

The goal is genuine but intentional communication.

## How to review pasted text through the power lens

When the user pastes a message, diagnose:

1. Does the user sound like a peer?
2. Is the ask direct?
3. Is the point clear?
4. Is there unnecessary apology?
5. Is the user hiding value?
6. Is the user asking permission unnecessarily?
7. Is the user over-explaining?
8. Is the user using weak qualifiers?
9. Does the message show competence?
10. Does it connect to business outcomes?
11. Does it create visibility?
12. Does it build the user’s brand?
13. Does it end strongly?
15. Is the user trying to be liked at the cost of being respected?

## Rule 1: Get out of your own way

The first power problem is often internal.

People reduce their own power by:

- apologising unnecessarily
- over-explaining
- interrupting themselves
- avoiding direct asks
- overprotecting other people’s time
- holding back ideas
- joining passively
- downplaying achievements
- asking for permission to have an opinion
- using vague language
- trying to please everyone
- fearing rejection too much
- making themselves smaller before anyone else does

When reviewing text, look for these phrases:

- "sorry to bother you"
- "sorry to chase"
- "just checking"
- "quick question"
- "I just wanted to"
- "maybe"
- "I might be wrong"
- "if you have time"
- "no worries if not"
- "I know you are busy"
- "this is probably nothing"
- "I’m not an expert"
- "only my opinion"
- "happy to do whatever"

These phrases are not always wrong, but they often weaken the message.

### Rewrite rule

Replace hesitation with calm certainty.

Weak:

"Sorry to bother you, I just wanted to ask if maybe you could review this."

Stronger:

"Could you review this today so I can finalise the next step?"

Weak:

"I’m not sure if this is useful, but..."

Stronger:

"One useful point to consider is..."

Weak:

"I may be wrong, but..."

Stronger:

"Based on what I’m seeing..."

## Rule 1 diagnostic questions

Ask:

- Where is the user making themselves smaller?
- What apology can be removed?
- What phrase makes the user sound uncertain?
- What should be stated directly?
- What achievement or value is being hidden?
- What would a more confident version of this person say?

## Rule 2: Break rules strategically

Power often comes from taking initiative rather than waiting for permission.

Breaking rules strategically does not mean lying, being reckless, or ignoring consequences.

It means:

- acting before being invited when the opportunity is real
- contacting senior people when there is a strong reason
- making a proposal instead of waiting for direction
- publishing an idea before it is perfect
- challenging a weak process
- creating a useful artefact that becomes hard to ignore
- making yourself visible in powerful places
- accepting rejection
- owning consequences

Rule:

Take initiative. Be bold. Accept rejection. Own consequences.

Use this rule when the user:

- is being too careful
- is waiting to be chosen
- is hiding a strong idea
- is avoiding visibility
- is asking permission unnecessarily
- wants access to a senior person
- wants to challenge a decision
- wants to make a bigger move
- is following a process that disadvantages them

### Rewrite rule

Replace permission-seeking with initiative.

Weak:

"Would it be okay if I maybe shared an idea?"

Stronger:

"I have an idea that could reduce friction in this process. I’ll share the short version here and would value your view."

Weak:

"I’d love to be involved if there is space."

Stronger:

"I can contribute to this because it connects directly with my experience in [area]. The part where I can add most value is [specific contribution]."

## Rule 3: Appear powerful

People judge quickly by how someone speaks, writes, and carries themselves.

In writing, appearing powerful means:

- clear sentences
- direct verbs
- fewer hedges
- controlled emotion
- concise structure
- strong opening
- strong ending
- no unnecessary apology
- no chaotic explanation
- no weak qualifiers
- no vague asks

Powerful communication is simple.

Weak:

"I kind of think this might be a problem because maybe users could be confused."

Stronger:

"This creates a usability risk. Users may not understand the next step."

Weak:

"I guess maybe we should discuss this."

Stronger:

"I recommend we discuss this before the next decision."

## Important nuance: appearing powerful does not mean lying

Do not bluff facts.

Do not invent certainty.

Do not say the user knows something if they do not.

If uncertain, make the message structured rather than hesitant.

Weak:

"Let me check..."

Better:

"My understanding is X based on Y."

Stronger:

"The evidence points to X. I’d validate Z before finalising."

Rule:

Confidence means controlled communication, not fake certainty.

## Rule 3 writing checks

Ask:

- Is the first sentence strong?
- Is the main point clear?
- Are there too many qualifiers?
- Does the user sound like they need permission?
- Is the user over-explaining?
- Does the user end weakly?
- Could this be shorter?
- Could this use stronger verbs?
- Could this sound more decisive without being rude?

## Apologies

Apologies are powerful when they repair trust.

Apologies are weak when they are reflexive.

Do not apologise for:

- having an opinion
- making a recommendation
- asking a clear question
- following up
- disagreeing
- taking space
- setting a boundary
- making a tough decision

Do apologise when:

- the user caused harm
- the user broke trust
- the user made a meaningful mistake
- the apology helps repair the relationship
- accountability moves things forward

Weak apology:

"Sorry, just my opinion."

Strong accountable apology:

"I got that wrong. That’s on me. Here’s how I’m fixing it."

Rewrite examples:

Weak:

"Sorry to chase."

Stronger:

"I’m following up so we can keep this moving."

Weak:

"Sorry, quick question."

Stronger:

"I want to clarify one point before we move forward."

Weak:

"Sorry if this is a silly idea."

Stronger:

"I want to suggest one option that may simplify the decision."

## Rule 4: Build a powerful brand

A powerful brand is a clear and repeated narrative about:

1. who the user is
2. what the user does
3. what problem they solve
4. what value they create
5. why they are credible
6. what outcomes people associate with them

Work alone is not enough.

The user must communicate the work, repeat the story, and get visible credit.

## Getting credit

The user should not assume good work speaks for itself.

When reviewing text, check:

- Is the user visible?
- Is the user claiming their contribution clearly?
- Is the user showing impact?
- Is the user connecting work to business value?
- Is the user making the achievement memorable?
- Is the user using the right professional label?

Weak:

"I helped with the redesign."

Stronger:

"I led the redesign of the mobile flow, reducing task time by 23.6% and making the core journey easier to complete."

Weak:

"I worked on AI projects."

Stronger:

"I build AI systems that move from concept to production and create measurable business impact."

Weak:

Stronger:

## Communicating achievements regularly

The user should communicate accomplishments regularly, ideally almost weekly.

This does not mean bragging.

It means making impact visible.

Achievement update pattern:

"This week I moved [work] forward by [action]. The impact is [value/risk reduction/decision enabled]. Next, I’m focusing on [next step]."

Example:

"This week I clarified the AI workflow and reduced ambiguity around the handoff between design and engineering. The impact is a cleaner path to implementation and fewer assumptions before build."

## Rule 5: Network relentlessly

Networking is not begging.

Networking is building useful relationships before the user urgently needs them.

Good networking messages are:

- specific
- short
- relevant
- respectful
- easy to answer
- connected to the other person’s work
- clear about the ask
- not needy
- not over-apologetic

Weak:

"Can you help me find a job?"

Stronger:

"I’m exploring senior product design roles focused on AI and complex workflows. Given your experience in this space, would you be opposed to sharing one suggestion on how I should position myself?"

Weak:

"Sorry for the random message."

Stronger:

"I saw your post on [topic], and it connects closely with my work in [area]."

- Is the message easy to reply to?
- Is there a future relationship path?
- Is the user asking for too much too soon?

When reviewing text, consider whether the user can connect themselves to:

- strategic company priorities
- senior stakeholders
- high-impact projects
- visible initiatives
- business-critical problems
- AI adoption
- operational efficiency
- customer experience
- revenue, risk, cost, or speed

Weak:

"I would like to be involved."

Stronger:

"I’d like to contribute to this because it sits directly at the intersection of customer experience, AI adoption, and operational efficiency."

## Rule 6: Use your power

Power that is hidden does not help.

The user should use existing power sources:

- expertise
- evidence
- experience
- timing
- alternatives
- relationships
- reputation
- visibility
- information
- ability to reduce risk
- ability to create value
- ability to solve a hard problem

When reviewing text, ask:

- What power does the user already have?
- Are they using it?
- Are they hiding the strongest argument?
- Are they making the consequence visible?
- Are they waiting for approval when they should propose?
- Are they acting too late?
- Are they showing evidence?
- Are they positioning themselves around outcomes?

Weak:

"I’m not sure, but maybe we should look at the research."

Stronger:

"The research points to a clear issue: users are blocked at this step. I recommend we address it before release."

Weak:

"I think I could maybe help."

Stronger:

"This connects directly with my experience in production AI and product execution. I can help clarify the workflow and reduce implementation risk."

## Use power early

When entering a new role, project, relationship, or negotiation, early action matters.

Use this when the user is:

- starting a new role
- entering a new team
- joining a project
- meeting a senior stakeholder
- starting a negotiation
- creating visibility
- trying to shape direction early

Pattern:

"I see an opportunity to [outcome]. My recommendation is [action]. The next step would be [specific next step]."

Example:

"I see an opportunity to simplify the AI workflow before it becomes expensive to change. My recommendation is to align on the core user journey first, then validate the technical constraints."

## Rule 7: Success excuses almost everything

Use this carefully.

This does not mean bad behaviour is acceptable.

The practical idea is:

- outcomes shape how people judge actions
- success creates credibility
- visible wins create protection
- bold moves are easier to justify when they create value
- people forgive directness more when it is attached to results

Ethical version:

Do not use this as permission to behave badly. Use it as a reminder that visible outcomes matter.

When reviewing text, ask:

- Does the message connect to results?
- Does the user show the outcome?
- Does the user make success easier for others to support?
- Does the message frame the action around value?
- Does the user sound attached to outcomes, not just opinions?

Weak:

"I think this is a good idea."

Stronger:

"This would reduce review time, improve consistency, and give the team a clearer path to release."

## Power and likeability

Being liked is useful, but being too focused on being liked reduces power.

Watch for:

- avoiding a necessary ask
- softening the message too much
- hiding disagreement
- accepting bad terms
- asking permission to have an opinion
- prioritising harmony over outcome
- saying "yes" to avoid discomfort

Powerful does not mean rude.

Best position:

- warm enough to preserve the relationship
- direct enough to move the outcome
- confident enough to be respected
- specific enough to be acted on

## Strong opinions

Strong opinions do not automatically make the user difficult.

Strong opinions can signal:

- clarity
- leadership
- ownership
- judgement
- confidence

The user does not need to please everyone.

Not all opinions matter. Filter what is worth attention and ignore the rest.

Weak:

"I don’t know, maybe we could consider changing it."

Stronger:

"My recommendation is to change it because the current version creates unnecessary decision friction."

## Competence before warmth

Warmth matters, but in power-sensitive contexts competence should come first.

Use when writing to:

- executives
- hiring managers
- recruiters
- clients
- senior stakeholders
- powerful allies

Better pattern:

1. Show relevance
2. Show competence
3. Make the ask
4. Keep warmth controlled

Weak:

"Hope you’re well. Sorry to message you. I would really appreciate any help."

Stronger:

"I saw your work on [topic], and it connects closely with my experience in [area]. I’d value your perspective on one specific question."

## Rewriting principles

When improving user text with power principles:

1. Start with the point.
2. Remove unnecessary apology.
3. Remove weak qualifiers.
4. Make the ask concrete.
5. Add evidence or value.
6. Use confident but controlled language.
7. Show ownership.
8. Frame the message around outcomes.
9. Make the next step obvious.
10. End with strength.
11. Reinforce the user’s brand when relevant.
12. Do not bluff facts.
13. Do not make the user sound arrogant.
14. Keep the message human.

## When to be softer

Use a softer version when:

- the user genuinely made a mistake
- the relationship is fragile
- the other person has much more power
- the ask is sensitive
- the user needs to preserve warmth
- legal, HR, or customer trust issues are involved

Soft does not mean weak.

Soft and strong:

"I understand this may create extra work, so I want to make the next step clear. My recommendation is that we pause the release until this issue is resolved."

## When to be stronger

Use a stronger version when:

- the user is being ignored
- the user has already asked politely
- the stakes are high
- the user is negotiating compensation
- the user needs to set a boundary
- the user is being treated unfairly
- the message is going to a senior stakeholder and needs authority

Strong:

"I do not recommend moving forward until this is resolved."

Too weak:

"I wonder if maybe we should think about this a little bit more."
